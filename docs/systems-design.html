① User Query

User types a natural language question. The agent sends it to claude-sonnet with all 8 tool schemas attached. Claude decides which tool(s) to call.

② Tool Selection

LangGraph's ReAct loop calls create_react_agent. Claude decides which tool(s) to call. MultiServerMCPClient routes each call to the correct MCP server — local stdio or remote HTTP.

③ MCP Dispatch

Server receives JSON-RPC call, dispatches to the matching tool function. Feed tools use asyncio.gather for concurrent item fetches — 2 RTTs instead of N+1.

④ Response & Memory

Tool results flow back through the LangGraph graph. MemorySaver checkpoints state after each turn — the agent remembers context across the full interactive session. Final answer printed via rich or saved to file / JSON.

Integration Assertions (3 tests)

🔌
All 8 tools registered
Spawns server.py as subprocess, connects via StdioServerParameters, calls list_tools(). Asserts the exact set HnPulse_GetTopStories … HnPulse_GetShowHn — no more, no less. Catches silent registration failures that unit tests can't see.
📝
Every tool has a description > 10 chars
Iterates all registered tools and asserts tool.description is non-empty and meaningful. A missing or trivial description causes Claude to mis-select tools — caught here before it reaches an eval or prod.
📐
All schemas are valid type: object
Asserts each tool's inputSchema is present and has "type": "object" at root — the MCP protocol requirement. A malformed schema causes silent tool-call failures in any MCP client.

Design Trade-offs & Decisions

Every meaningful choice — what was picked, what was rejected, and why.

⚡
Concurrent item fetches via asyncio.gather
Performance
Chose asyncio.gather(*[fetch(id) for id in ids]) — all item fetches fire simultaneously in a single async event loop turn.
Skipped Sequential for id in ids: await fetch(id) — simple but stacks RTTs linearly. 10 items = 10× latency.
Why HN's Firebase API returns only ID arrays from feed endpoints. Every item needs a separate HTTP call. With gather, latency collapses from N+1 RTTs → 2 RTTs (~300ms vs ~3s for 10 stories). In an agentic loop this compounds — a slow tool breaks perceived responsiveness.
Trade-off Slightly higher burst connection count to Firebase. Acceptable — Firebase is public-CDN-backed and we cap at 30 items max.
🏗️
Tools as pure functions, not @app.tool decorators
Architecture
Chose Plain async def functions in tools/. Framework wiring done once in server.py via app.add_tool().
Skipped @app.tool decorator on each function — couples every module to a live MCPApp instance at import time.
Why Unit tests call tool functions directly — no MCPApp construction, no framework bootstrap, no circular imports. Tests run in ~0.3s flat. If tools were decorated, each test module would pull in the full server context, adding overhead and fragility.
Trade-off Slightly more wiring code in server.py. Worth it — this is the same separation used by FastAPI (router vs handler) and Django (url patterns vs views).
✂️
Algolia response cleaning before returning to LLM
Context Efficiency
Chose Strip _highlightResult, children, _tags, updated_at from every Algolia hit before returning.
Skipped Return raw Algolia JSON — easiest path but sends ~2KB per result to the LLM.
Why For 20 results: ~36KB → ~4KB (9× token reduction). Matters in a multi-turn agentic loop where search may be called several times. Reduces cost and keeps the model's attention on signal, not HTML fragments and raw comment ID arrays.
Trade-off Stripping happens at the tool layer — callers that wanted raw highlight data can't get it. Accepted: no LLM use-case needs _highlightResult HTML.
🔒
stdio transport over HTTP/SSE
Security & Transport
Chose Agent spawns MCP server as a subprocess; communication over stdin/stdout using JSON-RPC 2.0.
Skipped HTTP transport (app.run(transport="http")) — requires binding a port, managing auth tokens, and handling network-level exposure.
Why No open ports. No credentials on the wire. Process lifecycle is managed by the agent — when the agent exits, the server exits. This is how Claude Desktop integrates MCP servers natively and is the safest model for local-first tools.
Trade-off Not shareable across multiple simultaneous clients. For multi-user deployment, HTTP + Arcade Deploy would be the right path.
🧪
Three-tier testing pyramid
Testing Strategy
Chose Unit (48, $0, ~0.5s) + Integration (3, $0, ~4s) + Evals (10, ~$0.002, ~15s). Unit tests include 12 error-scenario tests and 7 fetch tool tests. Evals use claude-haiku to validate Claude's tool-selection accuracy.
Skipped VCR cassettes / recorded HTTP fixtures — brittle to API schema changes and require re-recording when responses evolve.
Why pytest-httpx intercepts at the transport layer regardless of how AsyncClient was constructed — works cleanly with asyncio.gather. Evals catch model-layer regressions that no unit test can catch: does Claude pick the right tool for a given query?
Trade-off Evals require ANTHROPIC_API_KEY and are marked @pytest.mark.eval — skipped in CI by default with -m "not eval".
🔁
LangGraph ReAct over hand-rolled agentic loop
Architecture
Chose create_react_agent from LangGraph with langchain-mcp-adapters. MemorySaver checkpointer persists conversation state. MultiServerMCPClient bridges MCP tools into the LangGraph graph automatically.
Skipped Hand-written while True loop: ~60 lines of message accumulation, stop_reason checks, tool dispatch boilerplate — valid but re-invents what LangGraph already provides.
Why LangGraph is the framework ArcadeAI reference projects use. create_react_agent handles the loop, message accumulation, and tool dispatch. MemorySaver + thread_id gives the agent persistent session memory across turns for free — the hand-rolled loop had none.
Trade-off Heavier dependency footprint (langgraph, langchain-anthropic, langchain-mcp-adapters). Accepted — these are maintained, well-tested packages; the hand-rolled loop was maintenance burden with no state support.
🌐
Dual API strategy: Firebase + Algolia
Architecture
Chose Firebase for real-time ranked feeds & item detail; Algolia for full-text search. Each API routed to purpose-built tools.
Skipped Algolia-only — it supports search across everything but its feed endpoints don't reflect HN's real-time ranking algorithm faithfully.
Why Firebase is the authoritative source for live ranking; Algolia is indexed (slight delay) but has full-text. Using both gives the agent the right tool for each question type — trend queries vs. opinion/research queries.
Trade-off Two HTTP clients to manage. Mitigated by sharing a single httpx.AsyncClient instance per tool call scope.
🛠️
No caching layer (intentional deferral)
Performance · Future
Chose No TTL cache in this version. Every tool call hits the upstream API fresh.
Skipped In-memory TTL cache (e.g. 60s on feed IDs) — HN top stories change slowly; repeated agent calls within a session would benefit significantly.
Why Within the 6-hour constraint, adding a cache risked masking bugs during testing (stale data). Correctness first, optimisation second. The gather-based latency is already acceptable for demo use.
Trade-off Redundant upstream calls in multi-turn sessions. Next iteration: cachetools.TTLCache with 60s TTL on feed ID lists, 5m on item details.
💾
Conversation state via MemorySaver
Architecture
Chose LangGraph's MemorySaver checkpointer + UUID thread_id per session. Interactive mode reuses one thread — the agent remembers what it said earlier. One-shot mode creates a fresh thread per invocation.
Skipped No state — every turn starts from scratch. The hand-rolled loop had this limitation: context was only maintained within a single messages list in memory, lost on restart.
Why A research assistant that forgets your previous question is not useful in practice. MemorySaver adds full conversation memory at zero infrastructure cost — no database, no external service. Swappable for SqliteSaver for persistence across restarts.
Trade-off In-memory only — state is lost when the process exits. Accepted for v1; SqliteSaver is a one-line swap for durable persistence.
🔗
Multi-service: second MCP server for article fetching
Architecture
Chose Separate hn_fetch MCPApp in src/hn_extras/ with a single fetch_article tool. Opt-in via ENABLE_FETCH=1. MultiServerMCPClient connects to both servers simultaneously — local stdio or remote HTTP independently.
Skipped Adding fetch_article directly to hn_pulse/server.py — simpler but couples URL fetching to the HN server. Introduces httpx calls to arbitrary external URLs in a server that should only talk to HN APIs.
Why Separation of concerns: each server has one domain. The fetch server is deployable independently and reusable by other agents. MultiServerMCPClient makes connecting to N servers as simple as connecting to one.
Trade-off Two server processes to manage locally. Mitigated by Docker Compose for production deployment and ENABLE_FETCH flag to keep the default path simple.
🚢
HTTP transport + Docker for remote deployment
Security & Transport
Chose Both MCP servers expose --host 0.0.0.0 --port N HTTP transport. Agent switches from stdio to streamable_http automatically when MCP_SERVER_URL is set. docker-compose.yml deploys both as isolated containers with health checks.
Skipped stdio-only deployment — ties the server process lifecycle to the client. Not deployable to a remote machine or cloud environment.
Why Enables real multi-machine architectures: server on a cloud VM, agent on a laptop. Zero code changes — transport is environment-variable-driven. Docker gives reproducible deployment and health-checked startup ordering.
Trade-off HTTP transport has no built-in auth — the server is open on whatever port it binds to. Acceptable for private network / VPN use; add a reverse proxy with auth for public exposure.
